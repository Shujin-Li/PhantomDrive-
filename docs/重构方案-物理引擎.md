## 一、问题分析

### 1.1 架空原因链

```
【你输入】              【引擎】                【物理】               【显示】
   │                      │                       │                     │
   ▼                      ▼                       ▼                     ▼
键盘按下W ──┐      ┌──────────────────┐   ┌────────────────┐   ┌─────────────┐
            │      │  GameEngine      │   │ MCPhysicsWorld │   │ GameView    │
键盘按下S ──┼─────→│ handleInput() ──→│...│ [从未被调用]   │   │ [等待数据]  │
            │      │ [空壳]           │   │ step()         │   │             │
方向盘 ←───┘      │                  │   │ [从未被调用]   │   │             │
                 │ onGameLoopTick() ─┼───│ getPlayerState │──→│ render()    │
                 │ [没有物理调用]     │   └────────────────┘   └─────────────┘
                 └──────────────────┘
```

### 1.2 断点分析

| 断点 | 位置 | 问题 |
|------|------|------|
| **断点 1** | `handleInput()` → `???` | 键盘输入没有连接到任何地方 |
| **断点 2** | `onGameLoopTick()` → `MCPhysicsWorld` | 游戏循环没有调用 `step()` |
| **断点 3** | `MCPhysicsWorld` → `VehicleSensor` | 物理结果没有传给传感器 |

### 1.3 根本原因

**架构设计时没有完整实现数据流**：

```
DustRacing2D 的做法：
输入 → Car类 → 物理引擎(内嵌在Car里) → 位置更新

PhantomDrive 的做法（计划）：
输入 → GameMode.handleInput() → GameEngine → 物理引擎 → 传感器 → 评分

PhantomDrive 的现状（实际）：
输入 ──→ ??? ──→ 传感器[只能接数据，不参与计算]
            ↑
     handleInput 是空壳
     游戏循环没有物理调用
     物理引擎躺在那里没人用
```

## 二、解决方案对比

### 2.1 方案 A：简化物理模型

在 `GameEngine` 里直接算，不依赖 MCPhysicsWorld：

```cpp
void GameEngine::onGameLoopTick() {
    // 1. 从 InputHandler 读取控制
    float throttle = m_inputHandler->getThrottle();    // W/↑ = 1, S/↓ = 0
    float brake = m_inputHandler->getBrake();          // S/↓ = 1
    float steering = m_inputHandler->getSteering();    // A/D/←/→ = -1~1

    // 2. 简化自行车模型物理
    VehicleState& v = m_vehicleStates["player"];
    float acceleration = throttle * 100.0f - brake * 80.0f;
    v.speed += acceleration * deltaTime;
    v.speed = qBound(0.0, v.speed, 200.0);
    v.rotation += steering * 2.0 * (v.speed / 100.0) * deltaTime;
    v.position += QVector2D(cos(v.rotation), sin(v.rotation)) * v.speed * deltaTime;

    // 3. 更新传感器
    m_playerSensor->updateSpeed(v.speed);
    m_playerSensor->updateSteeringAngle(steering * 30.0);
    m_playerSensor->updateBrakeState(brake > 0);

    // 4. 更新显示
    emit m_gameView->vehiclePositionChanged(v.position, v.rotation);
}
```

**优点**：
- 改动量小，1-2 天可完成
- 不需要理解 MCPhysicsWorld 内部

**缺点**：
- 物理计算散落在 GameEngine 里，职责不清
- VehicleState 只是裸数据，没有行为
- 难以扩展（加入漂移、碰撞等复杂物理）

---

### 2.2 方案 B：引入 Car 类（推荐）

模仿 DustRacing2D 的架构，使用封装好的 Car 类：

```
DustRacing2D 的 Car 类
        ↓
    复制/继承到 PhantomDrive
        ↓
   PlayerCar : public Car    (键盘控制)
   AICar : public Car         (AI 控制)
        ↓
   GameEngine 管理 QMap<QString, Car*>
```

**优点**：
- 物理和状态封装在一起，职责清晰
- 每个 AI 对手也是一个 Car，代码一致
- 复用 DustRacing2D 已验证的 Car 类
- GameEngine 只负责调度，不涉及具体物理
- 易于扩展（漂移、碰撞、道具效果等）

**缺点**：
- 需要理解 Car 类的接口
- 需要处理好 Car 和 VehicleSensor 的关系

---

### 2.3 方案对比总结

| 对比项 | 方案 A（简化物理） | 方案 B（Car 类） |
|--------|-------------------|------------------|
| 工作量 | 1-2 天 | 2-3 天 |
| 代码质量 | 中等 | 高 |
| 可扩展性 | 低 | 高 |
| 复用 DustRacing2D | 否 | 是 |
| 职责分离 | 差 | 好 |
| AI 对手一致性 | 差（独立计算） | 好（都是 Car） |
| 风险 | 低 | 中低 |
| **推荐程度** | ⭐⭐ | **⭐⭐⭐⭐** |

## 三、重构后的架构

### 3.1 目标架构

```
┌──────────────────────────────────────────────────────────────────┐
│                         GameEngine                               │
│                                                                  │
│  ┌─────────────┐    ┌──────────────┐    ┌───────────────────┐   │
│  │InputHandler │    │   m_cars     │    │ VehicleSensorMgr  │   │
│  │ (键盘输入)   │───→│ QMap<Car*>   │───→│  (数据采集)       │   │
│  └─────────────┘    └──────────────┘    └───────────────────┘   │
│                           │                        │             │
│                           ▼                        ▼             │
│                    ┌──────────────┐         ┌───────────┐       │
│                    │   Car 类     │         │ScoreManager│       │
│                    │  - 物理计算   │         │ (评分系统) │       │
│                    │  - 位置/速度  │────────→│           │       │
│                    │  - 碰撞检测   │         └───────────┘       │
│                    └──────────────┘                            │
└──────────────────────────────────────────────────────────────────┘
```

### 3.2 数据流

```
键盘 / AI 决策
      │
      ▼
InputHandler / AIOpponent
      │
      ▼ setInput(throttle, brake, steering)
   Car 类
      │
      ├──→ update(deltaTime)  // 物理计算
      │
      ▼ getPosition() / getRotation()
GameEngine::onGameLoopTick()
      │
      ├──→ emit vehiclePositionChanged()  ──→ GameViewWidget (UI)
      │
      └──→ sensor->updateFromCar()  ──→ ScoreManager (评分)
```

## 五、具体改动示例

### 5.1 GameEngine（E-03 扩展）

> 原始任务：E-03 GameMode 抽象基类+模式框架（**A** 负责）
> 扩展任务：接入 Car 类 + InputHandler

```cpp
// GameEngine.h
class GameEngine : public QObject {
    Q_OBJECT
public:
    void onGameLoopTick(float deltaTime) override;

private:
    QMap<QString, Car*> m_cars;        // 替代 VehicleState
    InputHandler* m_inputHandler;      // 新增 (E-13)
    VehicleSensorManager* m_sensorMgr; // 已存在
};

// GameEngine.cpp
void GameEngine::onGameLoopTick(float deltaTime) {
    // 1. 玩家输入 → Car
    Car* player = m_cars["player"];
    player->setInput(
        m_inputHandler->getThrottle(),
        m_inputHandler->getBrake(),
        m_inputHandler->getSteering()
    );

    // 2. 更新所有 Car 的物理
    for (auto* car : m_cars) {
        car->update(deltaTime);

        // 3. 更新 UI
        emit vehiclePositionChanged(car->id(), car->position(), car->rotation());

        // 4. 更新传感器
        m_sensorMgr->updateFromCar(car);
    }
}
```

### 5.2 InputHandler（E-13 新增）

// InputHandler.h
class InputHandler : public QObject {
    Q_OBJECT
public:
    float getThrottle() const { return m_throttle; }

    float getSteering() const { return m_steering; }

    void keyPressEvent(QKeyEvent* event) override;
    void keyReleaseEvent(QKeyEvent* event) override;

private:
    float m_throttle = 0.0f;   // 0~1
    float m_brake = 0.0f;       // 0~1

void VehicleSensor::updateFromCar(Car* car) {
    updatePosition(car->position());

    updateVelocity(car->velocity());
    updateSteeringAngle(car->steeringAngle());
    updateSpeed(car->speed());
    updateBrakeState(car->isBraking());

## 八、结论

### 8.1 推荐方案

**使用方案 B（引入 Car 类）**，理由：

1. **时间可控**：10 天足够完成
2. **风险低**：复用 DustRacing2D 已有代码
3. **质量高**：职责分离，架构清晰
4. **易扩展**：未来加入漂移、碰撞等更简单
5. **影响小**：UI 组、评分组、交通组、道具组都不需要改

### 8.2 答辩时的说辞

> "我们采用了基于自行车模型的实时物理计算，参考了 DustRacing2D 的 Car 类架构。车辆控制完全解耦，UI、评分、交通系统都通过标准接口与车辆交互。"
