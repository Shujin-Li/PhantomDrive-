# A-B 驾驶评分算法与 AI 教练报告模块说明

## 1. 模块职责

A-B 模块负责将驾驶过程中的结构化数据转化为可解释、可展示、可联调的评分报告。模块核心目标包括：

1. 基于 `DrivingData` 与 `ViolationEvent` 计算驾驶总分；
2. 输出安全性、规则遵守、驾驶平顺性、效率等维度评分；
3. 生成结构化 `ScoreReport`，供 UI、AI 教练报告、Q-Learning 反馈等模块复用；
4. 通过 `ScoreManager` 发出 `scoreReady(ScoreReport)` 信号，方便后续与 U-A UI 层联调；
5. 提供离线 mock 版 `AIAPIClient`，将评分结果转化为中文教练报告；
6. 预留 `QLearningFeedback` 字段，方便后续 A-A 将驾驶表现转化为训练反馈信号。

当前版本定位为 **A-B v1 独立可测版本**。本轮没有修改主项目根目录 `CMakeLists.txt`，没有修改 `physics/MiniCore`，也没有把评分逻辑写入 `LearningMode`，而是新增独立 `scoring` 模块，保证低耦合、可回滚、便于后续联调。

## 3. A-B v1 调用链

A-B v1 的整体调用链如下：

```text
DrivingData / ViolationEvent
        ↓
TrafficRuleEnforcer（可选规则补充）
        ↓
DrivingScoreCalculator
        ↓
ScoreReport
        ↓
ScoreManager
        ↓
scoreReady(ScoreReport)
        ↓
AIAPIClient Mock
        ↓
中文 AI 教练报告
        ↓
QLearningFeedback
```

具体流程如下：

1. 外部模块可以直接调用：

```cpp
ScoreManager::evaluate(data, violations)
```

或通过数据采集接口调用：

```cpp
ScoreManager::evaluateFromCollector(collector)
```

其中 `collector` 走已有的 `IDrivingDataCollector` 接口。

1. 若设置了 `TrafficObjectManager`，`ScoreManager` 可以额外调用：

```cpp
TrafficRuleEnforcer::checkSequence(...)
```

对交通灯、限速、行人区域等规则进行补充检测。

1. `DrivingScoreCalculator::evaluate(...)` 负责核心评分逻辑，输出 `ScoreReport`。
2. `ScoreReport` 支持：

```cpp
toJson()
toMarkdown()
```

用于结构化传输、调试输出和报告展示。

1. `ScoreManager::generateCoachReport(...)` 调用 `AIAPIClient`，生成中文 Markdown 教练报告。
2. 成功评分后，`ScoreManager` 发出：

```cpp
scoreReady(report)
```

生成教练报告后，发出：

```cpp
coachReportReady(markdown)
```

若评分失败或输入为空，发出：

```cpp
scoringFailed(reason)
```

## 4. 评分维度

当前评分算法采用总分制，最终分数范围为 `0–100`，并根据总分生成等级：

```text
A：90–100
B：80–89
C：70–79
D：60–69
F：0–59
```

A-B v1 主要包含四个评分维度：

### 4.1 Safety Score：安全性评分

安全性评分重点关注驾驶过程中的高风险行为，包括：

- 碰撞；
- 行人碰撞或行人区域违规；
- 急加速；
- 急刹车；
- 其他可能体现安全风险的驾驶行为。

该维度主要反映车辆是否保持了安全距离、是否存在危险操作、是否对交通参与者造成风险。

### 4.2 Rule Compliance Score：规则遵守评分

规则遵守评分重点关注交通规则相关违规行为，包括：

- 超速；
- 闯红灯；
- 逆行；
- 限速区违规；
- 行人区域违规。

该维度主要用于衡量驾驶过程是否符合交通规则要求。

### 4.3 Smoothness Score：驾驶平顺性评分

平顺性评分关注驾驶操作是否稳定，包括：

- 急加速；
- 急刹车；
- 急转向；
- 转向角变化过大；
- 操作输入过于激进。

该维度用于衡量驾驶体验与操控稳定性，也可以为 AI 教练报告提供行为改进建议。

### 4.4 Efficiency Score：效率评分

当前版本中的效率评分较保守，主要基于数据完整性和平均速度合理性进行初步判断。由于真实赛道进度、路线效率、任务目标等信息尚未完全接入，暂不做复杂效率评价。

后续若 E-B 或 track 模块提供更完整的赛道进度、目标点、路线偏离、圈速标准等信息，可以进一步增强该维度。

## 5. 当前评分算法使用的字段

### 5.1 使用的 `DrivingData` 字段

当前评分算法主要使用以下字段：

```text
timestamp
speed
currentSpeedLimit
position
acceleration
steeringAngle
```

这些字段用于计算速度统计、超速比例、急加速、急刹车、急转向、限速区表现等指标。

### 5.2 使用的 `ViolationEvent` 字段

当前评分算法主要使用以下字段：

```text
timestamp
type
description
position
speedAtViolation
speedLimit
penaltyPoints
```

这些字段用于违规分类计数、扣分计算、报告解释和教练建议生成。

### 5.3 使用的 `ViolationType`

当前主要处理以下违规类型：

```text
Collision
SpeedOverLimit
RedLight
PedestrianCollision
WrongWay
```

这些类型会影响总分、分项评分、违规统计、教练建议和 Q-Learning 反馈。

## 6. ScoreReport 字段说明

`ScoreReport` 是 A-B 模块的核心输出结构。它不继承 `QObject`，定位为轻量数据结构，便于在评分、UI、报告生成和训练反馈之间传递。

### 6.1 基础字段

`ScoreReport` 至少包含：

```text
sessionId
vehicleId
generatedAt
totalScore
grade
summary
violations
metrics
breakdown
coachAdvices
qLearningFeedback
```

其中：

- `sessionId`：本次评分会话 ID；
- `vehicleId`：车辆 ID；
- `generatedAt`：报告生成时间；
- `totalScore`：总分，范围为 `0–100`；
- `grade`：等级，如 `A / B / C / D / F`；
- `summary`：整体总结；
- `violations`：本次驾驶中记录到的违规事件；
- `metrics`：驾驶统计指标；
- `breakdown`：分项评分与扣分明细；
- `coachAdvices`：面向用户的教练建议；
- `qLearningFeedback`：面向 A-A 的强化学习反馈数据。

### 6.2 ScoreMetrics

`ScoreMetrics` 记录驾驶过程中的统计指标，包括：

```text
dataPointCount
durationMs
averageSpeed
maxSpeed
collisionCount
speedViolationCount
redLightViolationCount
pedestrianViolationCount
wrongWayViolationCount
harshAccelerationCount
harshBrakingCount
sharpSteeringCount
overspeedFrameCount
overspeedRatio
```

这些指标主要用于解释分数来源，也可以被 UI 或调试日志直接展示。

### 6.3 ScoreBreakdown

`ScoreBreakdown` 记录分项评分和扣分明细，包括：

```text
safetyScore
ruleComplianceScore
smoothnessScore
efficiencyScore
totalPenalty
collisionPenalty
speedPenalty
redLightPenalty
pedestrianPenalty
smoothnessPenalty
```

该结构用于说明总分是如何由不同维度计算得到的，方便后续调参和展示。

### 6.4 CoachAdvice

`CoachAdvice` 用于保存结构化教练建议，包括：

```text
category
severity
message
evidence
```

其中：

- `category` 表示建议类别，如安全、规则、平顺、操控；
- `severity` 表示严重程度，如高、中、低；
- `message` 是中文建议文本；
- `evidence` 是建议依据，例如碰撞次数、超速比例、急刹次数等。

### 6.5 QLearningFeedback

`QLearningFeedback` 用于给后续 A-A 强化学习方向提供结构化反馈，包括：

```text
reward
normalizedScore
safetyRisk
ruleCompliance
collisionPenalty
speedPenalty
smoothnessPenalty
terminalPenalty
recommendedActionHint
```

其中：

- `reward`：综合奖励值；
- `normalizedScore`：归一化分数，通常为 `totalScore / 100`；
- `safetyRisk`：安全风险，范围为 `0–1`；
- `ruleCompliance`：规则遵守度，范围为 `0–1`；
- `collisionPenalty`：碰撞惩罚；
- `speedPenalty`：速度相关惩罚；
- `smoothnessPenalty`：操作平顺性惩罚；
- `terminalPenalty`：严重违规或终止类事件惩罚；
- `recommendedActionHint`：建议动作提示，例如提前刹车、保持车距、控制速度等。

## 7. scoreReady 信号说明

`ScoreManager` 是 A-B 模块对外组织评分流程的核心类，继承 `QObject`，负责把输入数据转化为 `ScoreReport`，并通过 Qt 信号对外通知。

核心信号包括：

```cpp
void scoreReady(const PhantomDrive::ScoreReport& report);
void coachReportReady(const QString& markdown);
void scoringFailed(const QString& reason);
```

### 7.1 scoreReady

当评分流程成功完成后，`ScoreManager` 会发出：

```cpp
scoreReady(report)
```

该信号后续可以由 U-A UI 层连接，用于展示：

- 总分；
- 等级；
- 分项评分；
- 违规列表；
- 教练建议；
- Q-Learning feedback 调试信息。

### 7.2 coachReportReady

当调用 `generateCoachReport(...)` 并成功生成 Markdown 教练报告后，`ScoreManager` 会发出：

```cpp
coachReportReady(markdown)
```

该信号后续可以用于 UI 展示完整 AI 教练报告。

### 7.3 scoringFailed

当输入为空、数据异常或评分流程无法正常完成时，`ScoreManager` 会发出：

```cpp
scoringFailed(reason)
```

当前版本对空数据做了保护，避免因无数据输入导致程序崩溃。

## 8. AIAPIClient Mock 说明

当前 `AIAPIClient` 是离线 mock 实现，不接真实网络 API，不读取 API key，不依赖外部模型服务。

当前职责是：

1. 接收 `ScoreReport`；
2. 根据总分、等级、风险指标和教练建议生成中文 Markdown 报告；
3. 保持接口形式稳定，方便后续替换为真实 LLM API；
4. 保证测试和展示时即使没有网络也能稳定运行。

当前 mock 报告一般包含：

```text
总览
重点建议
强化学习反馈
TODO：未来接入真实 LLM API
```

目前不接真实 API 是为了保持 A-B v1 的稳定性。真实 API 会引入网络请求、异步回调、鉴权、超时、失败重试、模型响应格式解析等复杂问题。当前阶段更重要的是先保证评分算法、报告结构、信号联调和测试链路稳定。

后续真实 API 接入时，建议保留 mock 模式，并增加 API 模式：

```text
AIAPIClient
  ├─ Mock mode：离线、稳定、适合测试与课堂展示
  └─ API mode：读取环境变量或配置文件，调用真实大模型生成教练报告
```

## 9. Q-Learning Feedback 说明

A-B 当前已经在 `ScoreReport` 中预留了 `QLearningFeedback`，用于后续对接 A-A 的 Q-Learning 或其他强化学习模块。

当前 Q-Learning feedback 的设计重点是：将驾驶表现从“人类可读评分”进一步转化为“训练可用信号”。

当前字段包括：

```text
reward
normalizedScore
safetyRisk
ruleCompliance
collisionPenalty
speedPenalty
smoothnessPenalty
terminalPenalty
recommendedActionHint
```

其中：

- `reward` 可以作为初步训练奖励；
- `normalizedScore` 表示整体驾驶表现；
- `safetyRisk` 表示安全风险；
- `ruleCompliance` 表示规则遵守程度；
- `terminalPenalty` 可用于严重事故、行人碰撞等终止类惩罚；
- `recommendedActionHint` 可作为调试或策略解释字段。

当前版本的 Q-Learning feedback 仍然是 v1 设计，后续需要与 A-A 进一步确认：

1. A-A 是否需要按帧 reward，还是只需要 episode-level reward；
2. 是否需要 state/action/reward/next_state/done 格式；
3. 是否需要将违规事件映射成不同训练惩罚；
4. 是否需要输出 JSON 文件或实时信号；
5. 是否需要和真实训练循环打通。

## 10. 当前限制

当前 A-B v1 已完成独立核心闭环，但仍存在以下限制：

1. **未接真实 UI。**
   当前只通过 example 测试验证 `scoreReady(report)` 信号触发，尚未与 U-A UI 页面联调。
2. **未接真实 LLM API。**
   `AIAPIClient` 当前为 mock 实现，只生成本地模板化中文 Markdown，不走网络。
3. **未接真实 Q-Learning 训练回路。**
   当前只输出 `QLearningFeedback` 字段，尚未与 A-A 的训练模块对接。
4. **未强依赖根目录主库构建。**
   当前 A-B 测试通过独立 example 构建，避免触发 `physics/MiniCore` 中尚未解决的构建问题。
5. **真实引擎数据链仍需联调。**
   当前评分算法基于已有 `DrivingData`、`ViolationEvent` 和模拟测试数据。后续需要接入 E-B 提供的真实车辆状态、碰撞状态、限速状态、交通对象状态流。
6. **效率评分仍较保守。**
   当前缺少完整赛道目标、路线偏离、任务完成度等信息，因此效率评分没有做复杂建模。
7. **TrafficRuleEnforcer 仍是基础封装。**
   当前版本主要基于已有 `TrafficObjectManager` 做规则补充检测，后续可以根据 E-B 的交通对象接口进一步增强去重、区域判断和事件触发逻辑。

## 11. 后续对接需求

### 11.1 对接 E-B / 引擎组

后续希望 E-B 提供更稳定的数据输入接口，包括：

```text
车辆实时位置
车辆速度
车辆加速度
转向角
刹车/油门状态
碰撞状态
碰撞对象 ID
当前限速
限速区 ID
交通灯状态
行人区域状态
赛道进度
路线偏离信息
```

这些信息可以进一步提升评分算法的准确性，尤其是安全性、规则遵守和效率评分。

### 11.2 对接 U-A / UI 组

后续 U-A 可以连接：

```cpp
scoreReady(const PhantomDrive::ScoreReport& report)
coachReportReady(const QString& markdown)
scoringFailed(const QString& reason)
```

UI 层可以展示：

```text
总分
等级
四维评分
违规列表
扣分明细
AI 教练建议
Markdown 报告
Q-Learning feedback 调试字段
```

建议 UI 初版优先展示 `ScoreReport::toMarkdown()` 或 `AIAPIClient` 生成的 Markdown 报告，降低联调复杂度。

### 11.3 对接 A-A / Q-Learning

后续需要与 A-A 确认：

```text
reward 的计算方式
是否需要逐帧 reward
是否需要 episode summary
是否需要 done / terminal 标记
是否需要 state-action-reward 格式
是否需要 JSON 导出
是否需要实时信号推送
```

当前 `QLearningFeedback` 已提供初步字段，但最终格式应根据 A-A 的训练输入要求调整。

### 11.4 后续真实 LLM API 接入

若项目后续需要真实 AI 教练报告，可以在现有 `AIAPIClient` 基础上增加 API 模式。建议后续接入时考虑：

```text
API key 安全存储
环境变量或配置文件读取
异步网络请求
超时处理
失败重试
无网络降级到 mock
模型响应 JSON 解析
Markdown 输出格式约束
```

真实 API 应只负责语言生成和解释，不应负责最终分数计算。最终分数仍应由 `DrivingScoreCalculator` 保持确定性和可复现性。
